# PROMPT PARA CURSOR AI - FLOW DE PREFECT PARA MIGRACION MYSQL A POSTGRESQL

## CONTEXTO
Necesito un flow de Prefect en Python que ejecute una migración completa de datos desde MySQL a PostgreSQL. La migración incluye 13 tablas con relaciones de foreign keys y tipos de datos específicos.

## ARQUITECTURA
- **Origen**: MySQL (alert_prod database)
- **Destino**: PostgreSQL (public schema, alert_prod database)
- **Orquestación**: Prefect 3.x
- **Ejecución**: Servidor Prefect (cloud o self-hosted)

## TABLAS A MIGRAR (Orden de dependencias)
1. cuenta (base, no tiene dependencias)
2. descarga_fuente_anmat (independiente)
3. disposicion (independiente)
4. frontend_routes (independiente)
5. busqueda (FK → cuenta)
6. busqueda_historica (FK → cuenta)
7. disposicion_contenido (FK → disposicion, incluye campo embedding_vector de tipo public.vector)
8. email (FK → cuenta)
9. overlay (FK → cuenta)
10. alerta_generada (FK → busqueda, disposicion)
11. alerta_generada_historica (FK → busqueda_historica, disposicion)

## REQUISITOS FUNCIONALES

### 1. CONEXIONES
- Conectar a MySQL con credenciales (origen)
- Conectar a PostgreSQL con credenciales (destino)
- Validar conectividad antes de iniciar migración

### 2. MIGRACIÓN DE DATOS
- Leer datos de cada tabla en MySQL en chunks (para no sobrecargar memoria)
- Transformar tipos de datos específicos:
  - BIT(1) → BOOLEAN
  - LONGTEXT → TEXT
  - TINYBLOB → BYTEA
  - DATETIME(6) → TIMESTAMP
- Insertar datos en PostgreSQL respetando orden de FK
- Manejo de campos NULL correctamente

### 3. VALIDACIÓN
- Contar registros origen vs destino por tabla
- Validar integridad referencial (FK)
- Checksum/hash en tablas críticas
- Reporte de discrepancias

### 4. HANDLING DE ERRORES
- Registrar errores por tabla
- Crear tabla de errores con detalles (tabla, row, error_msg)
- Retry automático con exponential backoff para fallos transitorios
- Logging detallado en cada paso

### 5. IDEMPOTENCIA
- Permitir re-ejecución sin duplicar datos
- Opción para truncar destino antes de migrar
- Versioning de ejecuciones (run_id, timestamp)

### 6. CAMPOS ESPECIALES
- `embedding_vector` en disposicion_contenido es de tipo `public.vector` (pgvector)
  - Si es NULL, insertar NULL
  - Si tiene valor, se asume ya está en formato vector
  - No intentar convertir desde MySQL
- `fecha_creacion` en email tiene DEFAULT CURRENT_TIMESTAMP
  - Preservar valores existentes, no regenerar
- `GENERATED BY DEFAULT AS IDENTITY` en email
  - Respetar la secuencia existente

## ESTRUCTURA DEL CÓDIGO REQUERIDO

### Archivo: migration_flow.py

```
Imports necesarios:
- prefect
- prefect.tasks
- sqlalchemy
- pandas
- logging
- dataclasses
- enum
- typing

Componentes:

1. Config Dataclass
   - mysql_connection_string
   - postgres_connection_string
   - chunk_size (default: 10000)
   - truncate_destination (default: False)
   - retry_attempts (default: 3)
   - log_file_path

2. Enum MigrationStatus
   - PENDING
   - RUNNING
   - COMPLETED
   - FAILED

3. DataMigrationTask
   - validate_connections()
   - get_source_count(table_name) → int
   - get_destination_count(table_name) → int
   - migrate_table(table_name, chunk_size) → MigrationResult
   - validate_referential_integrity()
   - generate_migration_report()

4. Flow Principal: migrate_alert_prod()
   - Input: config (Config dataclass)
   - Tasks en orden:
     a) validate_connections
     b) For each table in ORDER:
        - migrate_table (con reintentos)
        - log_result
     c) validate_referential_integrity
     d) generate_migration_report
     e) send_notification (email o webhook)

5. Subflow: migrate_single_table()
   - Input: table_name, source_engine, dest_engine, config
   - Output: MigrationResult
   - Lógica:
     - Leer chunks de MySQL
     - Transformar tipos de datos
     - Insertar en PostgreSQL
     - Validar counts

6. Task Utilities:
   - transform_row_mysql_to_postgres(row, table_schema)
   - execute_with_retry(func, retry_attempts, backoff_factor)
   - log_migration_error(table, error, row_data)
```

## DETALLES TÉCNICOS

### Conversión de Tipos MySQL → PostgreSQL
```
BIT(1) → BOOLEAN (True/False)
TINYBLOB → BYTEA (bytes)
LONGTEXT → TEXT
DATETIME(6) → TIMESTAMP
INT → INTEGER
VARCHAR → VARCHAR
DATE → DATE
NULL values → preservar como None
```

### Manejo de Foreign Keys
- Deshabilitar constraint checks antes de migrar (PostgreSQL)
- Respetar orden de tablas (bases primero)
- Re-habilitar constraints al final
- Validar integridad referencial post-migración

### Logging
```
Nivel INFO: inicio/fin de tabla, counts
Nivel WARNING: campos NULL inesperados, tipos convertidos
Nivel ERROR: fallos de inserción con contexto
Estructura: timestamp | level | table | message | row_id
```

### Performance
```
Chunk size default: 10,000 registros
Batch inserts en PostgreSQL (ON CONFLICT DO NOTHING si existe)
Connection pooling en ambas DBs
Índices: mantener desactivados durante bulk insert, reactivar al final
```

## OUTPUTS ESPERADOS

1. **Log File**: migration_YYYYMMDD_HHMMSS.log
2. **Reporte CSV**: migration_report_YYYYMMDD_HHMMSS.csv con columnas:
   - table_name
   - source_count
   - destination_count
   - inserted_count
   - error_count
   - duration_seconds
   - status (success/partial/failed)

3. **Error Table en Destino**: migration_errors
   - migration_run_id
   - table_name
   - row_id
   - error_message
   - error_timestamp

4. **Notification**: Email o webhook con resumen:
   - Total tables: X
   - Successful: Y
   - Failed: Z
   - Total records migrated: W
   - Duration: HH:MM:SS

## DEPLOYMENT EN PREFECT

```
1. Crear work pool (local o remote)
2. Registrar flow: prefect deployment build migration_flow.py -n prod
3. Apply deployment: prefect deployment apply migration_flow/prod-deployment.yaml
4. Trigger manualmente o en schedule

Alternative: Ejecutar directamente con:
prefect run migration_flow.py --mysql-host X --postgres-host Y
```

## CONSIDERACIONES ESPECIALES

### embedding_vector
- Campo en disposicion_contenido de tipo `public.vector` (pgvector)
- Preservar valores NULL
- No intentar procesar/transformar valores (son embeddings existentes)
- Usar COPY o bulk insert directo, no parsear

### Datos Sensibles
- No loguear passwords o credentials
- Usar Prefect blocks para almacenar credenciales
- Auditar acceso a migration_errors table

### Rollback
- Opción para recrear schema destino desde cero
- Script de cleanup: DROP cascades si es necesario
- Checkpoint de validación: si falla FK integrity check, ROLLBACK

## EJEMPLO DE USO

```python
from migration_flow import migrate_alert_prod, MigrationConfig

config = MigrationConfig(
    mysql_connection_string="mysql+pymysql://user:pass@localhost:3306/alert_prod",
    postgres_connection_string="postgresql://user:pass@localhost:5432/alert_prod",
    chunk_size=10000,
    truncate_destination=False,
    retry_attempts=3,
    log_file_path="/var/log/migrations/"
)

# Ejecutar
result = migrate_alert_prod(config)
print(f"Migration status: {result.status}")
print(f"Report: {result.report_path}")
```
